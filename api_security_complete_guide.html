Chapter 01 Foundations

Application Architecture &
Threat Surface

Before we can secure a system, we must understand its shape. The architecture of an application determines how data flows, where trust boundaries exist, and crucially — where attackers will focus their energy.

Architecture Patterns

Modern applications are built on a spectrum of architectural patterns, each carrying its own security profile. Understanding these patterns is not academic exercise — it directly determines which attacks are possible and which defences are appropriate.

🏛️
Monolithic Architecture

The traditional three-tier model — Presentation, Application, Persistence — deployed as a single unit. Security controls are centralised, which simplifies some aspects of defence. However, this creates an "all eggs in one basket" risk: a compromise of the application tier can directly expose database credentials. Patch cycles are slow because the entire application must be rebuilt and redeployed for any single change.

🔬
Microservices Architecture

Decomposes applications into small, independent services organised around business capabilities. Each service runs its own process and communicates via lightweight APIs (REST or gRPC). The attack surface becomes dramatically larger — with numerous connection points each requiring authentication and authorisation. The upside is micro-segmentation: a compromised service has a more limited blast radius.

⚡
Serverless (FaaS)

Event-driven execution where cloud providers manage the underlying infrastructure. Code runs only when triggered by events. The ephemeral container model reduces malware persistence — attackers cannot easily maintain a foothold. The risk profile shifts toward over-permissioned IAM roles and "Denial of Wallet" attacks, where excessive function invocations generate catastrophic cloud bills.

☁️
SaaS & Hybrid

SaaS shifts almost all infrastructure responsibility to the provider — but customers remain responsible for identity management, data governance, and configuration. Hybrid architectures combining on-premises and cloud services introduce the hardest problem: maintaining consistent security policies across environments, with gaps at transition points being prime exploitation targets.

The API Security Gap

Traditional security tooling was designed for a world of server-rendered web pages. APIs represent a fundamentally different problem that existing tools handle poorly.

⚠️
Critical Gap

Traditional tools like SAST and DAST lack the context to test API-specific logic. SAST may catch generic code flaws but will miss issues like Broken Object Level Authorization (BOLA) — where the code is syntactically correct but logically wrong. DAST tests running applications but cannot fully interpret API request/response semantics, leading to systematic false negatives on the most dangerous API vulnerabilities.

Why Network Perimeters Fail APIs

Network firewalls inspect IP addresses and ports — they have zero visibility into application logic. A firewall cannot distinguish between a legitimate customer fetching their own order data and an attacker manipulating an order ID to access someone else's. Web Application Firewalls (WAFs) operate on a negative security model — they block known bad patterns — but are blind to API-specific logic and data contracts.

The "Headless" API Problem

APIs have no user interface. Where a web application might use a button that only appears for administrators, an API endpoint is equally accessible to any authenticated HTTP client. Attackers can interact directly with backend logic using raw HTTP tools like curl, Postman, or Burp Suite — bypassing every frontend restriction. Over-permissioned APIs give attackers broad access the moment a single identity is compromised.

A trust boundary is any point where data transitions from one zone of control or trust to another. These boundaries are where the majority of security controls must be applied — and where the majority of attacks succeed.

Under Zero Trust principles, treat the browser as fundamentally untrusted. Enforce Content Security Policy (CSP) and rigorous input validation. Any data arriving from the client — cookies, tokens, form fields — must be treated as potentially malicious.

Frontend → API

SPAs and mobile apps use JWTs for authentication. Tokens must not be stored in localStorage (vulnerable to XSS). Use HTTP-only, Secure, SameSite cookies or in-memory storage to reduce theft vectors.

Service-to-Service (East-West)

Internal microservice traffic is often implicitly trusted — this is catastrophically wrong. Zero Trust mandates explicit authentication using Mutual TLS (mTLS). Apply service-level authorisation with token claims to enforce least-privilege and limit lateral movement after a breach.

Third-Party Integrations

Blindly trusting third-party API data is a supply chain vulnerability. Any data arriving from an external partner must be treated as untrusted input — validated, filtered, and sanitised before processing or storage, regardless of how trusted the partner appears.

💡
The Overlapping Risk Reality

Modern breaches rarely exploit a single vulnerability in isolation. Attackers chain weaknesses: SSRF to steal cloud credentials, then BOLA to exfiltrate sensitive records at scale, then a webshell for persistence. Failure in one security domain provides a foothold to exploit deeper flaws. This is why defence-in-depth across all layers is non-negotiable.

Chapter 02 Protocol

HTTP Protocol
Deep Dive

HTTP is the substrate of the web. Security practitioners must understand it deeply — not just how it works when things go right, but how attackers exploit its design characteristics and how visibility degrades across protocol versions.

HTTP Fundamentals & Statelessness

HTTP/1.1 treats each request as entirely independent — the server retains no memory of previous interactions. This statelessness is architecturally elegant but creates immediate security implications.

How Applications Simulate State

Because HTTP is stateless, every mechanism for "remembering" a user is a layer built on top — and each is a potential attack vector. Applications use cookies, session IDs, or tokens to link requests to a user identity. If an attacker can steal or forge any of these, they inherit the user's session without ever knowing their password.

Attack Perspective
  • Session Hijacking: Steal a valid session cookie to impersonate the user
  • CSRF: Exploit automatic cookie attachment to forge authenticated requests
  • Token Theft: Extract JWTs from XSS-vulnerable localStorage
  • Cleartext Interception: Sniff credentials over HTTP (no TLS)
  • Replay Attacks: Re-send captured valid requests to repeat actions
Defence Perspective
  • HttpOnly Flag: Blocks JavaScript from reading cookies — defeats XSS theft
  • Secure Flag: Cookies only sent over HTTPS
  • SameSite=Strict/Lax: Limits cross-site cookie transmission — defeats CSRF
  • TLS Everywhere: Mandatory HTTPS prevents cleartext interception
  • Short Token Lifetimes: Limits the window a stolen token remains useful

HTTP Methods as Attack Signals

Standard end-users primarily interact with GET and POST. The presence of other HTTP methods — especially in unexpected contexts — is often a meaningful security signal.

Method Legitimate Use Attacker Use Risk Level
GET Read resources Parameter manipulation, enumeration Medium
POST Create resources, forms Injection payloads, mass assignment High
PUT Replace entire resource Overwrite data, privilege escalation via mass assignment High
DELETE Remove resources BFLA — deleting others' resources High
PATCH Partial resource update Selective field manipulation Medium

HEAD Check resource existence Stealth enumeration without body transfer Low

Status Codes as Reconnaissance Intel

HTTP status codes leak information about backend behaviour. Security teams should monitor status code distributions — anomalous patterns often indicate active attacks in progress.

404 Bursts

A sudden spike in 404 Not Found responses usually indicates automated scanning or forced browsing — an attacker enumerating endpoints, files, or directories that don't appear in the UI.

500 Spikes

Internal Server Errors triggered by specific requests often reveal successful probing — for example, a malformed SQL query triggering a database error that confirms injection vulnerability exists.

401 / 403 Patterns

High volumes of 401 Unauthorised suggest credential stuffing or brute force. High volumes of 403 Forbidden from an authenticated session suggest privilege escalation attempts.

The Confused Deputy — X-Forwarded-For

When requests pass through a CDN or load balancer, the backend sees the proxy's IP, not the real client. Proxies insert X-Forwarded-For to preserve the original source. But this header is entirely attacker-controlled. If a WAF or backend implicitly trusts it to determine client identity — for rate limiting, geo-blocking, or access control — an attacker trivially spoofs it to appear as a trusted internal address and bypasses all IP-based controls.

Protocol Evolution & Visibility Challenges

Each major HTTP version has introduced performance improvements that simultaneously create security blind spots for traditional monitoring tools.

Version Key Changes Security Impact
HTTP/1.1 Text-based, one request per connection Human-readable; easy to inspect but cleartext without TLS
HTTP/2 Binary framing, HPACK header compression, multiplexing (multiple concurrent requests over one TCP connection) Non-human-readable without tooling; multiplexing obscures flow analysis for legacy IDS/IPS; HPACK complicates header inspection
HTTP/3 Moves from TCP to UDP (QUIC protocol); TLS 1.3 mandatory and integrated into transport Encrypts payloads, packet numbers, and handshake metadata; nearly all network-level monitoring is blind; many enterprise firewalls cannot inspect QUIC

DoH illustrates these visibility challenges perfectly. DNS queries — traditionally cleartext on UDP port 53, fully visible to network monitoring tools — are now encapsulated inside encrypted HTTP/2 or HTTP/3 streams on port 443. To any network monitor, DoH traffic is indistinguishable from normal web traffic.

This means attackers can use DoH to bypass internal DNS security controls, resolve malicious domains without appearing in DNS logs, and establish command-and-control channels that blend into normal HTTPS traffic. Defenders can no longer rely on DNS visibility as an intelligence source without deploying endpoint-level DNS interception.

Where Inspection Must Now Happen

Given that TLS 1.3 with Perfect Forward Secrecy (PFS) prevents passive decryption even if the server's private key is later compromised, traditional network-based monitoring cannot decrypt traffic. Inspection must be performed at two types of authorised points:

  • Endpoint Detection (EDR): Inspect traffic at the host before encryption occurs. The application process can be monitored for anomalous outbound connections, suspicious process tree behaviour, or unexpected data writes.
  • Authorised TLS Termination Points: Deploy WAFs, load balancers, or API gateways as intentional TLS termination proxies that decrypt, inspect, and re-encrypt traffic. This is the only scalable approach for protecting web-facing APIs.

Tools for HTTP Traffic Analysis

Burp Suite / OWASP ZAP

Interception proxies that act as intentional Man-in-the-Middle points. Allow security testers to pause, modify, and replay HTTP/HTTPS requests. Essential for testing API logic, bypassing client-side restrictions, and understanding authentication flows. The primary tool for manual API security assessment.

Wireshark / tcpdump

Capture raw network traffic. Analysts perform stream reconstruction to reassemble fragmented TCP/IP packets into coherent conversations. Useful for protocol analysis and non-encrypted traffic, but increasingly limited as TLS 1.3 adoption grows.

Chapter 03 Architecture

Web Application Architecture
& Security Model

The three-tier web application architecture remains the conceptual backbone of modern web security — even as implementations have evolved dramatically. Understanding what each tier owns, and where security controls must live, is foundational.

The Three-Tier Model

🖥️
Presentation Tier

Web servers (Nginx, Apache, IIS) or modern SPA frameworks (React, Vue, Angular) running in the browser. Collects user input and presents output. Historically a thin proxy; now increasingly intelligent with significant logic running client-side. Must be treated as untrusted territory.

⚙️
Application Tier

Middleware and frameworks (Spring Boot, Django, Rails, Express). Executes business logic, enforces authorisation rules, validates inputs. The critical gatekeeper — the only tier permitted to communicate directly with the database. Must never be directly internet-accessible.

🗄️
Persistence Tier

SQL databases (MySQL, PostgreSQL, Oracle), NoSQL stores (MongoDB, DynamoDB), or object storage (S3, Azure Blob). Increasingly managed by cloud providers — shifting the security responsibility model significantly toward configuration and access control rather than OS-level hardening.

Security Implications of Tier Isolation

Tier isolation — physical or logical separation between tiers — enables granular network-layer access controls, firewalls, and IDS between each layer. Critically, it provides compromise containment: breaching the presentation tier doesn't automatically grant database access. An attacker must pivot through the application tier's authorisation logic to reach sensitive data.

⚠️
Collapsed Tiers — The Two-Tier Antipattern

Combining presentation and application layers — often done to reduce hosting costs — drastically increases risk. Compromising the web server simultaneously exposes application logic and database credentials. All three tiers can be breached in a single step. This pattern is extremely common in legacy PHP applications and misconfigured cloud deployments.

The Evolution of SPAs and the Expanding Attack Surface

Modern Single Page Applications (SPAs) built on React, Angular, or Vue have shifted the MVC pattern — traditionally a server-side responsibility — directly into the user's browser. The browser is now effectively an extension of the server application.

This has profound security implications. The entirety of the application's frontend source code is delivered to the untrusted client environment. Adversaries can scrutinise this code to identify:

  • Hidden API endpoints — internal or administrative endpoints referenced in JavaScript but not exposed in the UI
  • Privilege-level logic — code that runs conditionally for administrators but is delivered to all users
  • Business logic — workflow constraints enforced client-side that can be bypassed by making direct API calls
  • Hard-coded values — API keys, environment identifiers, or internal service URLs accidentally bundled in production builds
⚡
The Cardinal Rule of Modern Web Architecture

The frontend is strictly an API consumer. Users can intercept, modify, and replay any request using proxies like Burp Suite. All client-side security controls must be treated as UX-only — they provide zero security guarantees. The server must independently verify every request's authentication, authorisation, and data integrity, treating all incoming client data as untrusted and potentially malicious.

Same-Origin Policy & CORS

The Same-Origin Policy (SOP) is one of the most important security mechanisms in the browser. It prevents malicious scripts on one site from reading sensitive data from another.

What Defines an "Origin"?

An origin is a precise three-part tuple: Protocol + Host + Port. All three must match exactly for two resources to be considered the same origin. The following pairs are different origins despite sharing a domain:

URL A URL B Same Origin? Reason
https://example.com http://example.com No Different protocol
https://example.com https://api.example.com No Different host (subdomain)
https://example.com https://example.com:8080 No Different port
https://example.com/a https://example.com/b Yes Only path differs — path is irrelevant

CORS — Controlled Relaxation of SOP

Modern applications frequently need to fetch data across origins (e.g., a frontend on app.company.com calling an API on api.company.com). CORS allows servers to explicitly declare which origins may access their resources via HTTP headers.

🚨
The Wildcard CORS Misconfiguration

Setting Access-Control-Allow-Origin: * permits any origin to read responses. This is particularly dangerous when combined with Access-Control-Allow-Credentials: true — though note that browsers actually block this specific combination per the CORS spec. The dangerous pattern is servers that dynamically reflect the incoming Origin header without validation, effectively creating a wildcard that does allow credentials. An attacker-controlled site can then make authenticated requests on behalf of the victim and read the response.

Preflight Requests

For "non-simple" requests (using methods like PUT or DELETE, or custom headers), browsers send an automatic OPTIONS preflight request first. The server must respond with appropriate Access-Control-Allow-Methods and Access-Control-Allow-Headers headers before the browser proceeds with the actual request. If the server doesn't grant permission, the browser blocks the transaction entirely — without the request ever reaching the backend handler.

TLS Architecture & the Zero Trust Problem

Where TLS is Typically Terminated

In most architectures, TLS is terminated at the load balancer or reverse proxy at the network perimeter. This centralises certificate management and offloads cryptographic processing from application servers. However, this creates a serious problem: East-West traffic (internal service-to-service communication) typically travels unencrypted across the internal network.

Traditional "castle-and-moat" security models considered this acceptable — the internal network was implicitly trusted. Zero Trust eliminates this assumption entirely. An attacker who breaches any internal service can now passively observe all internal API traffic, steal tokens, and move laterally without encountering any cryptographic barriers.

Service Meshes as a Solution

Service meshes like Istio or Linkerd solve East-West encryption by injecting sidecar proxies into every pod. These proxies transparently upgrade internal communications to Mutual TLS (mTLS) — where both client and server authenticate using PKI certificates before any data is exchanged. Neither the application code nor the developers need to implement this; it happens transparently at the infrastructure layer.

TLS 1.3 & Perfect Forward Secrecy

TLS 1.3 is mandatory for HTTP/3. PFS ensures that even if a server's long-term private key is later stolen, an attacker cannot decrypt previously captured sessions. Each session uses ephemeral key material that is discarded after the connection. This provides protection against retroactive decryption — critical for long-term confidentiality of sensitive API communications.

Downgrade Attack Risk

Supporting legacy protocols (SSLv3, TLS 1.0) makes systems vulnerable to downgrade attacks like POODLE and LOGJAM, where an adversary forces negotiation of weak export-grade encryption. Mandate TLS 1.2 minimum, prefer TLS 1.3, and use ECDHE cipher suites to ensure PFS. Disable all weaker suites at the server configuration level.

Chapter 04 Identity

Authentication
Fundamentals

Authentication is the process of verifying identity claims. Getting it wrong means an attacker can become any user in your system. This chapter covers the mechanisms, their strengths, and the sophisticated modern attacks that bypass even "strong" authentication.

Identity, Authentication & Authorization — Separating the Concepts

These three concepts are frequently conflated in code and architecture, creating dangerous logic flaws. They must remain strictly separated:

🪪
Identity

The unique representation of a subject — human, machine, or application — in a digital system. Defined by attributes in a directory (Active Directory, Entra ID). Identity answers "who are you claiming to be?" before any verification occurs.

🔑
Authentication (AuthN)

Verifying the truth of an identity claim. "Are you who you say you are?" Accomplished through four factors: something you know, something you have, something you are, and something you do. Multiple factors provide higher assurance.

🛡️
Authorization (AuthZ)

Decision logic that occurs only after successful AuthN. "What are you permitted to do?" Must follow least-privilege and deny-by-default. Successful authentication does NOT automatically imply any level of access.

💥
The Coupling Trap

When the same mechanism handles both authentication and authorisation, a breach in one model inevitably leads to a breach in the other. Many BOLA vulnerabilities exist because developers check "is this user authenticated?" but never ask "does this authenticated user own this specific resource?" — conflating AuthN with AuthZ.

Authentication Mechanisms

Form-Based Authentication

The most common mechanism — an HTML form submitting credentials via HTTP POST. Highly flexible, supports custom logic like MFA and CAPTCHA. Mandatory requirement: TLS must protect the form submission. POST requests transmit credentials in cleartext at the HTTP level; TLS is the only thing preventing interception.

HTTP Basic Authentication

A legacy built-in scheme where credentials are Base64-encoded and sent in the Authorization header. Critical misconception to avoid: Base64 is encoding, not encryption. It can be trivially decoded by any tool. Additionally, Basic Auth has no native logout mechanism — credentials remain cached in the browser until it is fully closed. Only acceptable when protected by TLS and even then should be replaced by modern alternatives.

Session Management

After authentication, the server assigns a cryptographically random Session ID with high entropy — designed to be computationally infeasible to guess. The server stores session data associated with this ID. The client includes the ID in subsequent requests, acting as a continuous re-authentication token. This approach requires server-side storage (database or cache) for session data, creating scaling considerations in distributed architectures.

Why SMS MFA is Deprecated (NIST 2016)

SMS-based OTP faces two classes of attack. SIM Swapping: an attacker socially engineers a mobile carrier into transferring the victim's phone number to an attacker-controlled SIM. All subsequent SMS messages, including OTP codes, go to the attacker. SS7 Interception: exploits fundamental vulnerabilities in the global telecommunications signalling protocol to intercept SMS messages in transit. These attacks are not theoretical — they have been used in high-profile financial fraud cases.

TOTP Vulnerability: Real-Time Phishing (AiTM)

TOTP codes change every 30–60 seconds but remain valid within that window. Adversary-in-the-Middle (AiTM) attacks use proxy tools like Evilginx to sit between the victim and the legitimate login page, acting as a real-time relay. The attacker captures the username, password, and current TOTP code — then immediately relays them to the real service. The victim successfully authenticates; the attacker receives a valid session cookie and replays it in their own browser. The entire attack happens faster than the TOTP window expires.

MFA Fatigue (Prompt Bombing)

Push notification MFA is vulnerable to a purely psychological attack. An attacker who has obtained valid credentials repeatedly triggers MFA push notifications — often dozens per hour, frequently late at night. The goal is to exhaust the victim's patience until they approve the request to silence the alerts. This technique was used in the 2022 Uber breach. Modern mitigations include number matching (displaying a code on the login screen that must be matched in the app) and geographic context.

FIDO2 — The Un-phishable Standard

FIDO2 and hardware tokens (YubiKeys) or platform authenticators (Passkeys on modern devices) use asymmetric cryptography. The hardware device is cryptographically bound to the specific domain during registration. When authenticating, the device signs a challenge from the server — but only if the domain in the browser matches the registered domain exactly. A perfectly cloned phishing site will have a different domain; the hardware key detects this and refuses to sign. AiTM attacks are technically defeated because the attacker's proxy domain will never match.

Browser Pivoting — Post-Exploitation MFA Bypass

Once an attacker has a foothold on a victim's machine via malware, they can use frameworks like Cobalt Strike to inject a proxy into the user's active browser process. The attacker routes their own web requests through the victim's browser, inheriting all existing authenticated sessions and cookies. Because the server sees requests originating from the already-authenticated browser, no new MFA challenge is triggered. This is why endpoint security and EDR are critical components of an MFA strategy — the cryptographic controls are sound, but the endpoint trust is assumed.

Legacy Protocol MFA Bypass

Protocols like IMAP, POP3, and SMTP were designed before MFA existed. They simply cannot challenge for a second factor during authentication. If these protocols are enabled on a mail server protected by MFA for the web portal, attackers use tools like MFASweep to identify endpoints where legacy authentication is still permitted — and authenticate directly, bypassing MFA entirely. The fix is to disable all legacy authentication protocols and enforce modern authentication universally.

Chapter 05 Identity

OAuth 2.0, OIDC
& JSON Web Tokens

Modern authentication at scale is built on delegated authorization frameworks. Understanding OAuth 2.0 and JWT deeply — including their vulnerabilities — is essential for any API security practitioner.

OAuth 2.0 — Delegated Authorization

OAuth 2.0 solves the access delegation problem: allowing a third-party application to access a user's resources without the user sharing their credentials. The classic analogy is a valet key — it grants limited access (drive the car) without full access (the boot, the glove box).

OAuth issues a time-bound Access Token restricted to specific scopes. The application never sees the user's password. The user authenticates directly with the Authorization Server (Identity Provider), which issues tokens to the application.

The OAuth 2.0 Flows

Flow Use Case Security Status
Authorization Code + PKCE All user-facing apps (web, mobile, SPA) Gold Standard
Client Credentials Machine-to-machine, no user present Correct for M2M
Implicit Flow Legacy SPAs Deprecated
Resource Owner Password (ROPC) Legacy systems Deprecated

Authorization Code Flow + PKCE — Why It's Gold Standard

The flow separates the browser from final token acquisition via a multi-step back-channel exchange. A short-lived authorization code (valid ~60 seconds) is returned to the browser — not the token itself. This prevents token leakage through browser history, referrer headers, or server logs. The code can only be exchanged for tokens via a direct server-to-server call.

PKCE (Proof Key for Code Exchange) prevents code interception attacks. At flow initiation, the client generates a random code_verifier, hashes it to create a code_challenge, and sends the challenge with the authorization request. When exchanging the code for tokens, the client must provide the original verifier. If an attacker intercepts the authorization code, they cannot exchange it without the verifier they never received.

⚠️
Why Implicit Flow is Deprecated

The Implicit Flow returned tokens directly in the URL fragment (e.g., #access_token=eyJ...). Tokens appear in browser history, server access logs, and the Referer header of subsequent requests. Any XSS vulnerability could exfiltrate the token from the URL. ROPC is deprecated because it requires users to give their password directly to the application — violating the core principle that users authenticate exclusively with the IdP, not individual apps.

OpenID Connect (OIDC) — Identity on Top of OAuth

OAuth 2.0 handles authorization ("what can you do?") but not authentication ("who are you?"). OIDC adds a standardised identity layer. The IdP issues an ID Token — a cryptographically signed JWT containing claims about the authenticated user: their unique identifier (sub), name, email, and the time of authentication. The application verifies this token using the IdP's public key.

OIDC is the foundation of modern Single Sign-On. Once a user authenticates with the IdP (Google, Microsoft, Okta), all OIDC-integrated applications trust that assertion without requiring re-authentication.

JSON Web Tokens — Structure & Security

JWTs are compact, self-contained tokens carrying identity and authorisation claims. They enable stateless authentication — APIs can validate requests without querying a central session store. A JWT has three Base64URL-encoded parts separated by periods: header.payload.signature. 

// Decoded JWT Header
{
  "alg": "RS256",   // Signing algorithm
  "typ": "JWT"
}

// Decoded JWT Payload (Claims)
{
  "iss": "https://auth.example.com",  // Issuer
  "sub": "user_7f3a9c",              // Subject (user ID)
  "aud": "api.example.com",          // Audience (intended recipient)
  "exp": 1735689600,               // Expiration (Unix timestamp)
  "iat": 1735686000,               // Issued At
  "scope": "read:orders write:profile"
}

Critical JWT Validation Checklist

  • Validate the signature — reject any token where signature verification fails
  • Hardcode the expected algorithm — never trust the alg header from the client
  • Reject the "none" algorithm — tokens with alg: none have no signature at all
  • Check exp — expired tokens must be rejected; missing check enables replay attacks
  • Validate aud — token for Service A must not work on Service B
  • Validate iss — token must come from the expected authorization server
  • Never store PII in payload — the payload is only Base64URL encoded, not encrypted

JWT Attack Techniques

Algorithm Confusion Attack

In RS256, the server signs with a private key and verifies with a public key. An attacker changes the header to HS256 (symmetric) and uses the server's public key (which is public by definition) as the HMAC secret. Poorly implemented libraries that trust the client's alg header will verify the attacker's forged signature successfully. Fix: hardcode the expected algorithm server-side.

"None" Algorithm Vulnerability

Some JWT libraries accept tokens where alg is set to none, meaning no signature verification occurs. An attacker decodes a legitimate JWT, modifies the payload (e.g., changes role to admin), sets alg: none, removes the signature section, and submits it. The library accepts it as valid. Some bypasses use case variants: nOnE, NONE.

Expired Token Replay

If the server doesn't validate the exp claim, a stolen token remains valid indefinitely — even after a user changes their password. The window of opportunity for an attacker extends to infinity. Short token lifetimes (5–15 minutes for access tokens) are the primary mitigation.

Audience Confusion

A token issued for a low-security internal "analytics" service can be replayed against a high-security "payments" service if both trust the same issuer and neither validates the aud claim. Strict audience validation is required on every service.

The "Achilles' heel" of JWTs: unlike server-side sessions that can be deleted from a database immediately, a signed JWT remains cryptographically valid until its exp claim passes — even if the user changes their password, or the token is known to be compromised.

Workarounds include: token blocklists (store revoked JTI claims in Redis until they expire — requires a database lookup per request, partially defeating the stateless advantage); short-lived tokens (access tokens of 5–15 minutes paired with longer-lived refresh tokens); or token families (refresh token rotation with reuse detection).

Symmetric vs Asymmetric Signing

HS256 — Symmetric (HMAC)
  • One secret key used for both signing and verification
  • Computationally fast and efficient
  • Secret must be shared with every service that verifies tokens
  • If any downstream service is compromised, the shared secret is exposed
  • Any service with the secret can also forge new tokens
  • Suitable only for single-service architectures
RS256 / ES256 — Asymmetric
  • Private key (AuthN server only) for signing; public key for verification
  • Public key can be freely distributed via JWKS endpoint
  • Downstream services cannot forge tokens — they only have the public key
  • Compromise of one downstream service doesn't compromise the signing key
  • Performance cost: asymmetric operations are ~100x slower than HMAC
  • The correct choice for distributed microservices architecture
Chapter 06 Identity

Authorization Models
& Access Control

Authorization is the most exploited category of vulnerability in modern APIs. Understanding the models, their failure modes, and how to implement them correctly is critical for any security-conscious developer or architect.

Access Control Models

RBAC — Role-Based

Permissions assigned to static roles (Admin, Editor, Viewer). Users assigned to roles. Simple to manage; updating a role changes permissions for all users with that role. Weakness: static nature lacks granularity — leads to "role explosion" when fine-grained control is needed. Cannot express "user can only edit their own documents."

ABAC — Attribute-Based

Access decisions made by evaluating rules against four attribute categories: subject (job title, clearance), resource (classification, owner), action (read/write), and environment (time of day, location). Extremely flexible and granular. Weakness: complex to implement; significant computational overhead; hard to audit.

ReBAC — Relationship-Based

Models users and resources as nodes in a graph; access is determined by relationships (edges). If User A owns Folder X and Document Y is inside Folder X, User A inherits access through the relationship. Used by Google Zanzibar (powers Google Drive). Handles complex hierarchical ownership naturally.

BOLA — Broken Object Level Authorization

BOLA (also called IDOR — Insecure Direct Object Reference) is consistently ranked #1 in the OWASP API Security Top 10. It occurs when an API verifies that a user is authenticated but fails to verify that the authenticated user owns or is permitted to access the specific resource being requested.

Root Cause: The "Authenticated = Authorized" Fallacy

The fundamental logic error is that developers use Object-Relational Mappers (ORMs) that bind user-supplied input directly to database queries — a user provides an ID, the ORM fetches that ID — without an intervening ownership check. The developer assumes "if a user is authenticated, any ID they provide must be theirs." This assumption is catastrophically wrong.

💡
Why UUIDs Are Not a Fix

Some developers attempt to mitigate BOLA by replacing sequential integer IDs with UUIDs, relying on obscurity to prevent enumeration. This fails because UUIDs can be leaked through other API responses (e.g., search results, notifications, or shared links), log files, or API documentation. A UUID-based BOLA vulnerability is harder to discover but equally exploitable. The only fix is authorization logic that verifies ownership — not ID format.

Correct Defence Pattern

Every request to a resource must be authorised against the user's identity stored in the server-side session or verified JWT claims. The application must never trust an ID provided by the client without confirming that identity owns or is permitted to access that resource. Modern architectures use centralised authorization middleware or policy engines to enforce these checks consistently across all endpoints.

BFLA — Broken Function Level Authorization

While BOLA is about accessing the wrong data, BFLA is about executing the wrong functions. A standard user gains the ability to invoke administrative endpoints — delete accounts, modify system settings, or promote themselves to admin. This is vertical privilege escalation.

Forced Browsing

Attackers use automated spidering or known admin path patterns to discover endpoints not linked in the UI: /api/admin/users, /api/v1/debug, /api/internal/config. The API handles these requests correctly — but any authenticated user who discovers them can invoke them if RBAC isn't enforced at the function level.

HTTP Method Manipulation

An API may correctly restrict DELETE /users/:id to admins, but fail to check roles for PUT /users/:id on the same endpoint. Switching the HTTP method bypasses the incomplete authorization check. This is common when authorization logic is applied piecemeal per method rather than centrally.

Parameter Manipulation

Some implementations use request parameters for privilege: submitting admin=true in a POST body, or changing URL case. These rely on security by obscurity — the developer "hid" the feature but didn't enforce authorization. Any API proxy tool instantly reveals and allows modification of these parameters.

Security by Obscurity

Hiding admin features from the UI creates a false sense of security. The server-side attack surface remains fully accessible. Attackers use Burp Suite, network traffic analysis, or source code inspection to discover these features. Obscurity is not security — every function must enforce its own authorization requirements.

Why Authorization Bugs Persist

Authorization vulnerabilities are uniquely difficult to catch through automation for several reasons:

  • Context-dependent logic: Authorization rules are unique to each application's business domain. A scanner cannot know that "sub-users should not see the parent account's billing data" without understanding the application's intent.
  • Syntactically valid requests: When authorization fails, the application often processes the request successfully and returns a 200 OK. An automated tool sees a valid transaction — it cannot distinguish authorized access from unauthorized access.
  • Happy-path testing: QA processes focus on verifying that features work correctly for authorized roles. Negative testing — verifying that unauthorized users are blocked — is frequently omitted or implemented superficially.
  • Microservices fragmentation: Authorization logic spread across dozens of loosely coupled services creates inconsistency. A single service may fail to validate a token's scope, providing an entry point that bypasses the entire authorization architecture.
Chapter 07 Vulnerabilities

Injection
Attacks

Injection vulnerabilities occur when untrusted data is sent to an interpreter as part of a command or query. Despite being well-understood for decades, injection remains one of the most prevalent and damaging vulnerability classes.

SQL Injection

SQL injection allows attackers to insert malicious SQL syntax into input fields, manipulating the query logic executed against the database. The consequences range from authentication bypass to complete data exfiltration or deletion. 

// Vulnerable query — string concatenation
query = "SELECT * FROM users WHERE username='" + input + "' AND password='" + pwd + "'"

// Attacker input: username = admin'--
// Resulting query — comment truncates the password check:
SELECT * FROM users WHERE username='admin'--' AND password='anything'

// Correct implementation — parameterized query
query = "SELECT * FROM users WHERE username=? AND password=?"
params = [input, pwd]  // Parameters treated as data, not SQL

ORM Tools — Partial Protection

Object-Relational Mapping tools have significantly reduced SQL injection risk by abstracting database interaction. However, vulnerabilities are frequently reintroduced when developers bypass the ORM for performance reasons — writing raw SQL queries, using string concatenation inside ORM hooks, or using "raw" query modes. Any raw SQL string that incorporates untrusted input is vulnerable.

NoSQL Injection

NoSQL databases (MongoDB) use JSON/BSON query structures. Attackers inject MongoDB query operators to manipulate logic: 

// Vulnerable MongoDB authentication check
// Attacker sends: {"username": "admin", "password": {"$ne": null}}
// The $ne (not equal) operator returns true for any non-null value
// Result: first admin record returned without knowing the password
db.users.find({
  username: "admin",
  password: { $ne: null }  // Injected operator bypasses auth
})

OS Command Injection

Occurs when an application passes unsanitised user input to a system shell. The application server itself becomes a weapon — executing arbitrary OS commands with the application's system privileges.

💀
Maximum Severity Impact

Successful OS command injection typically results in complete server compromise: reading arbitrary files (/etc/passwd, application source code, environment variables), establishing persistent webshells, exfiltrating credentials from memory or disk, and pivoting deeper into the internal network. Attackers use command separators like ; (Linux), & or | (both platforms) to chain malicious commands to legitimate ones.

GraphQL-Specific Injection Risks

GraphQL's Introspection feature — when left enabled in production — provides attackers with a complete map of all types, fields, queries, and mutations. This is equivalent to handing an attacker the entire API schema. Beyond information disclosure, GraphQL's flexible query model enables resource exhaustion attacks: deeply nested recursive queries (e.g., Authors → Books → Authors → Books…) can consume all server CPU and memory. Batching attacks pack hundreds of password guesses into a single HTTP request, bypassing rate limits that count requests rather than operations.

Chapter 08 Vulnerabilities

XSS, CSRF &
Client-Side Attacks

Client-side attacks target the browser itself — turning the user's trusted session against them. Understanding XSS, CSRF, and clickjacking at a mechanistic level is essential for implementing effective defences.

Cross-Site Scripting (XSS)

XSS occurs when untrusted data is included in web output without proper encoding, allowing attackers to execute arbitrary JavaScript in a victim's browser. The script runs with the full privileges of the victim's session.

Type 1: Stored XSS

Payload permanently stored on the server (database, comment field, profile). Served to every user who views the affected page. A single injection can silently impact thousands of users without requiring targeted phishing. The most severe form.

Type 2: Reflected XSS

Payload returned immediately in the server's response, typically via URL parameters. Requires social engineering — the victim must click a crafted link. Commonly delivered via phishing emails. The malicious script is "reflected" from the server back to the user's browser.

Type 0: DOM-Based XSS

Payload never reaches the server — it exists entirely in client-side JavaScript. The browser's DOM is modified using attacker-controlled data from "sources" (URL fragments, location.hash). Data flows through the DOM to "sinks" like innerHTML or document.write that execute it.

What Attackers Can Do With XSS

  • Session Hijacking: document.cookie exfiltrated to attacker-controlled server to impersonate the user without knowing their password
  • Credential Harvesting: Inject fake login forms that submit credentials to attacker infrastructure
  • Browser as Bot (BeEF): "Hook" the browser to a command-and-control server for keystroke logging, screenshots, internal network scanning
  • LocalStorage Theft: Access and exfiltrate JWTs and other tokens stored in localStorage
  • DOM Manipulation: Alter page content to deceive users into performing actions they don't intend

Why Input Filtering Alone Fails

Input filtering is unreliable as a primary defence. Attackers encode payloads using UTF-7, hexadecimal, or Unicode encoding to bypass filters. More importantly, input that is safe in one context (plain text) can become malicious in another (HTML attribute, JavaScript string, CSS). Context-aware output encoding at the point of rendering is the only reliable defence — data must be encoded according to the context where it appears.

⚡
Framework Escape Hatches

Modern frameworks like React provide secure-by-default output encoding. However, developers can bypass this protection using escape hatches: React's dangerouslySetInnerHTML, Angular's bypassSecurityTrustHtml, or Vue's v-html. Any use of these APIs with untrusted data reintroduces XSS. These APIs should be treated as high-risk code requiring mandatory security review.

Cross-Site Request Forgery (CSRF)

CSRF exploits the browser's automatic inclusion of cookies in requests. When a user is logged into bank.com, their session cookie is automatically attached to every request to bank.com — regardless of where the request originates. A malicious site can trigger a state-changing request to bank.com, and the server will process it as legitimate because it carries a valid session cookie.

Why CSRF Targets State-Changing Operations

CSRF cannot read the response (SOP prevents this), but it can trigger actions: fund transfers, password changes, account deletion, privilege changes. It's particularly dangerous for endpoints that trust cookies alone as proof of intent.

Clickjacking

An attacker overlays a transparent iframe containing the target application over a convincing decoy page. The user believes they're clicking a button on the decoy, but their click actually hits a sensitive function on the invisible target — approving a payment, changing a password, or granting permissions.

Defence relies on preventing the application from being embedded in iframes on untrusted sites. Use the CSP frame-ancestors 'none' directive (modern replacement for X-Frame-Options: DENY) to instruct browsers to refuse all iframe embedding. JavaScript framebusting scripts are unreliable — attackers can disable scripts or use the sandbox attribute to block them.

Chapter 09 Vulnerabilities

SSRF, XXE &
Deserialization

These three vulnerability classes are particularly dangerous because they allow attackers to use the application itself as a weapon — making the server perform actions on the attacker's behalf, often bypassing network security controls entirely.

Server-Side Request Forgery (SSRF)

SSRF occurs when a web application fetches a remote resource based on user-supplied input without validating that the destination is safe. The attacker coerces the application server into acting as a proxy — making requests to destinations it normally wouldn't reach. Because the request originates from the backend server itself, it bypasses perimeter firewalls that block external access to internal services.

Attack Targets

  • Internal APIs and databases: Services like Elasticsearch or Redis that are not authenticated because they're "safely" inside the network
  • Administrative interfaces: Kubernetes API server, Docker daemon, internal management portals
  • Cloud Instance Metadata Service (IMDS): Available at http://169.254.169.254 on all major cloud platforms. Returns IAM role credentials, security tokens, and environment configuration for the instance
🚨
Capital One Breach — SSRF in Practice

In 2019, an attacker exploited SSRF via a misconfigured WAF to query the AWS IMDS at http://169.254.169.254/latest/meta-data/iam/security-credentials/. The WAF's IAM role had overly broad S3 permissions. The attacker retrieved temporary credentials, used them to list and download contents from over 700 S3 buckets, and exfiltrated data from approximately 100 million customers. The entire attack chain: SSRF → credentials theft → data exfiltration.

  • Strict allowlists — only permit requests to explicitly approved hostnames/IP ranges
  • Disable HTTP redirections — prevent chaining through open redirects
  • Use IMDSv2 — requires a session token (PUT request) before metadata is accessible, defeating simple SSRF
  • Network egress filtering — application servers should not be able to reach internal subnets via outbound HTTP

XML External Entity (XXE) Injection

XXE targets applications that parse XML input with older or misconfigured parsers. The XML specification includes "entities" — essentially macros that the parser expands. External entities reference URIs that the parser fetches and includes in the document. Attackers abuse this to read arbitrary files or trigger SSRF through the XML parser. 

<!-- Attacker sends this XML payload -->
<?xml version="1.0"?>
<!DOCTYPE foo [
  <!ENTITY xxe SYSTEM "file:///etc/passwd">
]>
<user>
  <name>&xxe;</name>  <!-- Parser expands entity, includes /etc/passwd content -->
</user>

The "Billion Laughs" DoS attack uses recursive entity expansion to exponentially consume memory: 

<!DOCTYPE bomb [
  <!ENTITY lol "lol">
  <!ENTITY lol2 "&lol;&lol;&lol;&lol;&lol;&lol;&lol;&lol;&lol;&lol;">
  <!ENTITY lol3 "&lol2;&lol2;&lol2;&lol2;&lol2;&lol2;&lol2;&lol2;&lol2;&lol2;">
  <!-- Continues until entity expands to gigabytes -->
]>

Fix: Completely disable DTD processing and external entity resolution in the XML parser configuration. Modern parsers often disable these by default, but legacy compatibility settings may re-enable them.

Insecure Deserialization

Serialization converts complex in-memory objects into a byte stream for storage or transmission. Deserialization reconstructs the object. The vulnerability arises when an application deserializes untrusted data — the deserializer may follow instructions embedded in the serialized stream to instantiate arbitrary classes before the application logic can inspect the resulting object.

Privilege Escalation via Object Tampering

Attackers intercept serialized objects stored in cookies, hidden form fields, or URL parameters and modify their properties. In a PHP session cookie containing a serialized user object with role: "user", changing it to role: "admin" and re-encoding it grants administrative access if the application deserializes and trusts this data without integrity verification.

Gadget Chains — Remote Code Execution

"Gadget chains" are sequences of legitimate classes already available in the application's classpath (like Java's CommonsCollections library) that execute dangerous actions during the deserialization process. Attackers craft serialized payloads containing these chains to achieve arbitrary code execution. Tools like ysoserial automate gadget chain generation for Java; Python's pickle module similarly allows code execution via the __reduce__ method.

Cloud computing has fundamentally changed what organisations are responsible for securing. The Shared Responsibility Model defines where provider obligations end and customer obligations begin — and misunderstanding this model is a leading cause of cloud breaches.

The Shared Responsibility Model

⚠️

Many organisations believe that moving to the cloud transfers security responsibility to the provider. In reality: you can outsource data management, but you can never outsource legal liability. If your customer data is exposed due to a misconfigured S3 bucket with public read access, the regulatory consequences fall on you — not AWS.

Common Cloud Security Failures

Overly Permissive IAM

IAM roles and policies using wildcards (Action: "*", Resource: "*") grant far more access than needed. A compromised function or instance with an over-permissive role allows attackers to access all S3 buckets, any DynamoDB table, or the entire EC2 fleet. Least privilege must be applied per-role.

Public Storage Buckets

S3 buckets or Azure Blob containers accidentally configured with public read access remain a primary cause of mass data exfiltration. Cloud providers now have "Block Public Access" account-level settings — these should be enabled globally and exceptions require explicit business justification.

Open Security Groups

VMs with SSH (port 22) or RDP (port 3389) open to the entire internet (0.0.0.0/0) are immediately targeted by automated scanners. Shodan continuously indexes these. Management ports should only be accessible through a bastion host or VPN, with source IP restrictions.

Static Long-Lived Credentials

AWS access keys beginning with AKIA are static and have no expiry. These are high-value targets for attackers scanning GitHub repositories and package registries. Prefer instance profiles, execution roles, and temporary STS credentials. Audit for static keys using IAM Access Analyzer.

Zero Trust in Cloud Microservices

Traditional "castle-and-moat" security assumed internal network traffic was trustworthy. Cloud microservices environments make this assumption lethal — services span accounts, regions, and providers. Zero Trust mandates: never trust, always verify.

Mutual TLS (mTLS)

Both client and server present PKI certificates before any connection is established. Eliminates MITM attacks and impersonation between internal services. Neither service can communicate with the other unless both have valid, trusted certificates. The foundation of Zero Trust service-to-service authentication.

Service Mesh (Istio / Linkerd)

Automates mTLS by injecting sidecar proxies into pods. All internal traffic is transparently upgraded to mTLS without application code changes. Provides traffic observability, circuit breaking, and fine-grained authorisation policies at the infrastructure layer.

SPIFFE Identity

A standard for issuing verifiable identity documents to workloads in dynamic environments. Every service receives a cryptographic identity (SVID) regardless of IP address or hostname — addressing the challenge of dynamic cloud environments where instances are constantly created and destroyed.

North-South vs East-West

North-South: external traffic entering the cluster, managed by API gateways and ingress controllers. East-West: internal service-to-service traffic, the most under-protected layer. Micro-segmentation via network policies prevents a compromised service from freely communicating with unrelated workloads.

Chapter 12 Infrastructure

Containers, Kubernetes
& Serverless Security

Modern deployment platforms — containers, orchestration platforms, and serverless functions — each introduce unique security considerations that require dedicated attention beyond traditional application security.

Container Security

Containers are isolated processes sharing the host OS kernel — a larger attack surface than VMs, which each have their own kernel. Container security must be addressed across the entire lifecycle: image build, registry storage, and runtime execution.

Image Scanning

Container images are the "master copy" of the environment. Any vulnerability or embedded secret in the image is replicated in every running instance — and every future deployment. Security must be integrated into the CI/CD pipeline via automated scanning before images are pushed to registries. Tools like Trivy, Clair, and Snyk build a software bill of materials per image layer, cross-referencing against CVE databases.

Runtime Security Controls

  • Run as non-root (runAsNonRoot: true) — limits blast radius if container is compromised
  • Read-only root filesystem — prevents attackers from writing webshells or modifying binaries
  • Drop all capabilities — remove Linux capabilities not needed by the application
  • No privileged containers — privileged containers can escape to the host trivially
  • Immutable infrastructure — never patch containers in place; rebuild from a clean base image

Kubernetes-Specific Security

The Kubernetes API server is the cluster's central control plane — its exposure is equivalent to root access to every workload. In the 2018 Tesla breach, an unauthenticated Kubernetes dashboard exposed cluster secrets including AWS credentials, which were then used for cryptocurrency mining. Every management interface must be authenticated, network-restricted, and monitored.

⚡
Runtime Monitoring with eBPF

Static scanning cannot detect exploits that only manifest at runtime. Tools like Falco, Cilium, and KubeArmor use eBPF technology to monitor kernel-level activity across all containers in real-time. They detect anomalous behaviour — a container spawning a shell, writing to unexpected filesystem locations, or making unusual network connections — and can alert or automatically terminate the offending workload.

Serverless (FaaS) Security

Serverless functions present a different threat model: the infrastructure is managed by the provider, but application code, IAM configuration, and data handling remain the customer's responsibility.

Cold Starts, Warm Environments & Data Persistence

A "cold start" deploys a fresh container. Providers keep environments "warm" (AWS Lambda: ~11 minutes, GCP: ~3 minutes) for reuse. Any sensitive data written to /tmp persists until the container is destroyed — and may be accessible to subsequent function invocations. Avoid storing sensitive data in writable directories and treat the function environment as potentially shared.

Each Event Trigger is an Attack Surface

Functions can be triggered by HTTP requests, S3 uploads, database changes, SQS messages, or scheduled events. Each trigger represents a potential injection point — an attacker controlling an S3 bucket that triggers a processing function can craft a malicious object to exploit the function's parsing logic.

IAM Role Least Privilege

Functions execute under an assigned IAM execution role. If this role uses wildcards (s3:* on Resource: *), a compromised function hands the attacker full S3 access across the entire account. Each function should have a unique IAM role granting only the specific permissions required for its task — verified against actual usage via IAM Access Analyzer.

Environment Variable Exposure

Temporary credentials (AWS_SESSION_TOKEN, AWS_SECRET_ACCESS_KEY) are injected as environment variables. Command injection vulnerabilities in function code allow attackers to execute env or read from /proc/self/environ to extract these credentials. Mitigate by preventing command injection (no shell execution of user-controlled input) and using managed secret stores instead of environment variables for long-lived secrets.

Infrastructure as Code Security

IaC (Terraform, CloudFormation, ARM templates) enables rapid, repeatable infrastructure deployment. A single misconfiguration in a template can instantly propagate thousands of vulnerable assets across an enterprise. Security must be integrated into the IaC development process itself.

Hardcoded Credentials

AWS access keys, API tokens, and SSH keys accidentally embedded in Terraform variables or CloudFormation parameters. These are committed to version control, visible to anyone with repository access, and permanent in git history even after deletion.

Unencrypted Storage

S3 buckets, EBS volumes, and RDS instances deployed without encryption at rest. IaC scanners detect missing server_side_encryption_configuration blocks or storage_encrypted: false settings before deployment.

Permissive Networking

Security group rules allowing all traffic from 0.0.0.0/0 on management ports. IaC scanning tools (Checkov, tfsec, KICS) catch these patterns in CI/CD before they reach production — "shifting left" the security gate.

✅
Immutable Infrastructure

Immutable infrastructure means resources are never modified in place. When a patch or configuration change is needed, a new instance is created from a hardened "golden image" and the old one is destroyed. This eliminates configuration drift — where a system's state gradually deviates from its approved security baseline — and ensures every production environment exactly matches the approved specification.

Chapter 13 Defence

Secure API
Design Principles

Security must be designed into an API from inception — not added as an afterthought. This chapter covers the foundational principles that, when applied consistently, eliminate entire classes of vulnerabilities before code is written.

Principle of Least Privilege

Every process, user, and program must operate with the absolute minimum permissions necessary for its specific task. This is a design constraint, not a configuration step — it must be applied from the first line of the API specification.

Scoped Token Design

Clients receive tokens with the narrowest possible scope for their intended use. A mobile app with read-only requirements receives a read-only token — even if the authenticated user could perform write operations. Scope restriction happens at token issuance, not at the consuming service.

Whitelist-Only Field Updates

Mass Assignment prevention: explicitly define which fields a client may update via a whitelist. Any field not in the whitelist is silently ignored regardless of what the client sends. Never use framework "auto-bind" features with untrusted input without an explicit allowlist of acceptable fields.

Backend Data Filtering

APIs must filter and mask sensitive data at the backend before responding. Never return a full database object and rely on the UI to hide fields. An attacker with a proxy tool sees the full unfiltered response. Return only the fields the client legitimately needs for its specific view.

Token Forwarding Limits

In microservice chains, backend services must not forward the caller's original over-privileged token to upstream services. Each service-to-service call should use a narrowly-scoped token for that specific integration. If Service A calls Service B, Service B receives a token for its specific task — not the end user's full-access token.

Error Handling — The Information Leakage Problem

Verbose error messages are one of the most underestimated security risks. A stack trace containing the database type, ORM library version, file paths, and SQL syntax hands an attacker a detailed map of the backend architecture.

Pagination, Sorting & Resource Limits

Unconstrained query parameters allow attackers to trigger resource exhaustion without exploiting any code vulnerability — they simply make legitimate-looking requests that consume disproportionate resources.

  • "Big Integer" DoS: requesting ?limit=100000000 attempts to load the entire database in one query. Server-side pagination with enforced maximum page sizes prevents this.
  • Property Enumeration via sortBy: Accepting arbitrary column names in ?sortBy=internalField reveals hidden database columns by observing which values trigger errors versus which return sorted results.
  • GraphQL Query Depth: Recursive nested queries can exhaust CPU and memory. Enforce maximum query depth, complexity scoring, and execution timeouts.

API Versioning & Lifecycle Management

"Zombie APIs" — deprecated versions left running but unpatched — are unmonitored backdoors. The longer an old version runs, the more likely it contains vulnerabilities patched in current versions. A disciplined API lifecycle process is essential.

Sunset & Deprecation Headers

Per RFC 8594: the Sunset header signals when an endpoint will become unavailable; Deprecation timestamps when usage is discouraged. Clients can detect these headers and migrate automatically. Consumers get warning; operators get a migration timeline.

API Gateway Lifecycle Enforcement

The API gateway is the single choke point for version retirement. Programmatically disable deprecated endpoints at their sunset date — no manual intervention, no forgotten endpoints. Automated sunsetting closes the zombie API window permanently.

Chapter 14 Defence

Client-Side
Security

The browser is the most hostile environment in the security stack — code runs in untrusted territory, users can modify it, and attackers can inject into it. Browser security mechanisms are the last line of defence for client-side data.

Token Storage Trade-offs

localStorage / sessionStorage
  • Directly accessible via JavaScript — any XSS can exfiltrate tokens
  • Persists across browser sessions (localStorage)
  • Commonly used for JWTs due to developer convenience
  • Zero protection against XSS — single vulnerability exposes all stored tokens
  • OWASP recommendation: do not store sensitive tokens in localStorage
HttpOnly Cookies
  • Inaccessible to JavaScript — XSS cannot read the token
  • Automatic inclusion in requests creates CSRF risk
  • SameSite=Strict/Lax mitigates CSRF for same-site contexts
  • Does not protect against subdomain-based attacks
  • Best practice: HttpOnly + Secure + SameSite=Strict
✅
Backend-for-Frontend (BFF) Pattern

The most secure token management architecture moves all token handling to a server-side BFF component. The browser never receives or stores tokens. The BFF issues encrypted HttpOnly+SameSite cookies that serve as session references — the actual tokens live server-side. XSS attacks in the browser cannot exfiltrate what the browser never receives. This adds complexity but eliminates the entire class of client-side token theft.

Security Headers

Header Protection Key Configuration
Strict-Transport-Security Forces HTTPS; prevents SSL stripping attacks max-age=31536000; includeSubDomains; preload
Content-Security-Policy Restricts script sources; prevents XSS execution and clickjacking script-src 'self'; frame-ancestors 'none'; object-src 'none'
X-Content-Type-Options Prevents MIME sniffing — stops scripts disguised as images nosniff (only valid value)

Content Security Policy — In Depth

CSP is the most powerful and complex security header. It defines an allowlist of trusted sources for every type of resource the browser can load. Key directives:

  • script-src — controls JavaScript sources. Avoid 'unsafe-inline' and 'unsafe-eval'; these defeat the purpose of CSP. Use nonces or hashes for any required inline scripts.
  • connect-src — restricts where fetch/XHR/WebSocket requests can be sent. Prevents exfiltration of stolen data to attacker servers.
  • frame-ancestors — modern replacement for X-Frame-Options. Controls which domains can embed this page in an iframe. Use 'none' to prevent all framing.
  • default-src — fallback for directives not explicitly set. Set to 'self' as a conservative baseline.

Subresource Integrity (SRI)

Modern applications load JavaScript from external CDNs. If that CDN is compromised — or if the specific script version is replaced — malicious code runs on every page that loads it. This is the supply chain attack vector that hit the British Airways breach (Magecart attack injected card-skimming code into a third-party script).

SRI allows browsers to verify that a fetched resource matches a known cryptographic hash. If the file has been altered, the hash won't match and the browser refuses to execute it:

Chapter 15 Defence

Secrets
Management

A leaked credential in a cloud environment is functionally equivalent to a username, password, and MFA code combined. Proper secrets management is one of the highest-ROI security investments an organization can make.

Where Secrets Live — and Leak

Hardcoded in Source Code

The most dangerous antipattern. Secrets in code are visible to everyone with repository access, including contractors and open-source contributors. More critically, git history is permanent — deleting a file doesn't remove it from the commit history. Automated scanners (truffleHog, GitLeaks) find these within seconds of a push to GitHub.

Config & Backup Files

Files like wp-config.php, .env, database.yml, or config.json containing credentials. Backup files (.bak, .old, ~) accidentally left in public web directories are trivially discovered by automated scanners. Directory listing exposing these files is an immediate critical incident.

Client-Side Code

Mobile apps and JavaScript bundles are delivered to untrusted clients. Any API key embedded in an APK can be extracted by decompiling the application. Browser developer tools expose all JavaScript, including any credentials inadvertently included in frontend bundles. No secret should ever be placed in client-side code.

Verbose Logs ("Toxic Logs")

Authorization headers, session tokens, and API keys written to application logs during debugging. Stack traces containing environment variable dumps. Log aggregation systems become treasure troves of credentials if not carefully designed. Never log sensitive headers or authentication materials.

Managed Secret Stores

Services like AWS Secrets Manager, HashiCorp Vault, and Azure Key Vault provide the correct architecture for secrets management. Applications retrieve secrets dynamically at runtime via authenticated API calls — the sensitive data stays in memory and never touches disk in plaintext.

Centralization

A single encrypted repository eliminates "secrets sprawl" — credentials scattered across dozens of config files, environment variables, and repositories. Security teams can see and manage the complete secrets inventory from one location.

Audit Trails

Every secret access is logged: who accessed which secret, when, from which IP address. This is critical for forensic analysis after a compromise — analysts can determine exactly which secrets were accessed and whether they need to be rotated.

Automated Rotation

Regular credential rotation limits the window of opportunity for stolen credentials. A leaked database password that rotates every 24 hours is usable for at most one day. Managed stores handle rotation for both the stored credential and the protected resource (e.g., rotating both the Secrets Manager value and the actual database password simultaneously).

Short-Lived Credentials

Static IAM access keys (AKIA...) have no expiry. Prefer AWS STS temporary credentials (valid for minutes to hours) that force regular re-authentication. Short token lifetimes combined with rotation dramatically reduce the blast radius of any credential compromise.

The Business Impact of Credential Leakage

Authentication Bypass

A leaked cloud API key bypasses all authentication controls — including MFA. The attacker authenticates as the application identity itself, not as a user. Standard authentication monitoring won't flag this because the credentials are technically valid.

Lateral Movement

Credentials found in one application (e.g., a connection string in a config file) often provide access to other systems — internal databases, cloud control planes, or CI/CD pipelines. A single hardcoded root key can grant complete infrastructure access.

"Denial of Wallet" & Destruction

Attackers with administrative cloud credentials can spin up GPU instances for cryptocurrency mining — generating bills of thousands of dollars per day. At the extreme, they can delete all VMs, databases, and backups simultaneously, permanently destroying the business's digital existence. Code Spaces, a GitHub competitor, was destroyed this way in 2014.

Chapter 16 Defence

Logging, Monitoring
& Detection

Attackers spend an average of 200+ days in compromised networks before detection. The difference between discovering a breach in hours versus months is almost entirely determined by the quality of logging and monitoring infrastructure.

What to Log — The Five Ws

Every security-relevant log entry must answer: Who (authenticated identity, session ID) performed What action (HTTP method, resource accessed, data changed) from Where (source IP, User-Agent, geographic location) at When (precise UTC timestamp) with What result (success/failure, status code, bytes transferred).

Category Events to Log Security Value
Authentication Login success/failure, MFA attempts, token issuance, session creation/termination Detect brute force, credential stuffing, session hijacking
Authorization 403 Forbidden responses, privilege escalation attempts, admin function access Detect BOLA/BFLA exploitation, insider threat
Input Validation Rejected inputs, schema validation failures, rate limit triggers Detect fuzzing, reconnaissance, injection attempts
Data Access CRUD operations on sensitive resources, bulk downloads, export operations Detect data exfiltration, scraping attacks
System Events Service start/stop, configuration changes, dependency failures Detect tampering, unauthorized configuration changes
⚠️
Never Log — PII & Credentials

Logging sensitive data creates a secondary breach target. Passwords, authentication tokens, credit card numbers, and full API keys must never appear in logs. Apply data minimization: mask partial values where reference is needed (e.g., last 4 digits of card), use pseudonymization for personal identifiers, and sanitise request bodies before logging. Logs that contain sensitive data are themselves a GDPR compliance liability.

Detection Signals — Reading Attack Patterns

Traffic Anomalies

Consistent machine-speed request timing (bots don't breathe). Unusually high request rates from a single IP or user. "Beaconing" — periodic requests at regular intervals indicating C2 communication. Request volume 10x normal from a single user account.

Authentication Signals

"Impossible travel" — logins from two geographic locations physically impossible in the elapsed time. New device fingerprint or user agent for an established account. Unusual time-of-day access. High-volume failed authentication attempts (credential stuffing).

Scanning Indicators

404 burst (automated path enumeration). 500 spike (injection probing triggering errors). OPTIONS requests against API endpoints (capability discovery). Sqlmap or Hydra signatures in User-Agent strings. SYN scans (half-open connections that never complete handshake).

Exfiltration Signals

Single user account accessing data volumes far beyond their normal baseline. Sequential object ID enumeration with high success rate (BOLA exploitation). Bulk export operations during off-hours. Outbound data transfer volume anomalies.

Cross-Layer Correlation

Individual log sources tell incomplete stories. The power of security monitoring comes from correlating signals across layers:

  • Application + Database: Did a suspicious web request actually result in a backend database query? Confirming the full chain distinguishes probing (no DB hit) from successful exploitation (DB query executed).
  • Identity + Network: Correlating authenticated User IDs from application logs with source IPs from firewall logs attributes network traffic to specific individuals — enabling tracking of an identity across devices and locations.
  • Distributed Tracing: In microservices, a single Trace ID that persists through the entire request chain (load balancer → API gateway → service A → service B → database) allows analysts to reconstruct the complete path of a malicious request across all components.
⚡
Data Graves

Many organisations collect logs but never analyse them. Post-breach forensics routinely reveals that the attack was visible in logs for months, but the volume of noise made the signals impossible to detect manually. Centralise all logs in a SIEM or data lake with automated correlation rules, alerting thresholds, and anomaly detection. Logs that are never read are just compliance theatre.

Threat Modeling

Proactive identification of risks by decomposing the application and understanding its attack surface. Most effective when performed during the design phase — fixing architectural flaws early is orders of magnitude cheaper.

R Repudiation Non-repudiation User denies performing an action; logs insufficient to prove otherwise

Testing Methodologies

SAST — Static Analysis

Scans source code without executing it. Finds flaws early; points to exact lines of code. High false-positive rate. Cannot detect runtime configuration issues, logic flaws, or environment-specific bugs. Best for: injection patterns, hardcoded secrets, dangerous function calls.

DAST — Dynamic Analysis

Black-box testing against a running application from an external attacker perspective. Finds runtime issues: insecure cookies, authentication flows, CORS misconfigurations. Late in the lifecycle; cannot pinpoint specific vulnerable code lines. Best for: integration testing of running services.

IAST — Interactive Analysis

Agent embedded in the application runtime monitors code execution in real-time. Combines SAST's code visibility with DAST's runtime context. Dramatically lower false-positive rate. Language-specific — requires an agent built for each technology stack. Best of both approaches.

Software Composition Analysis (SCA) & SBOM

Modern applications consist of 70–80% third-party code. Every dependency inherits its upstream vulnerabilities. SCA tools automatically inventory all packages (including transitive dependencies — the libraries your libraries depend on) and cross-reference against the National Vulnerability Database. A Software Bill of Materials (SBOM) is a machine-readable inventory of every component — when a new CVE like Log4Shell is disclosed, an SBOM allows instant identification of affected services across the entire estate.

DevSecOps Culture

Security cannot remain a specialised siloed department that acts as a bottleneck. In DevSecOps, security teams transition from gatekeepers (blocking releases) to guardrail providers (enabling safe deployment by default).

  • Security Champions: Senior developers with security training embedded in engineering squads. They run threat modeling sessions, conduct security-focused code reviews, and translate compliance requirements into practical coding guidelines.
  • Secure-by-Default Infrastructure: New services automatically get HTTPS, network segmentation, and least-privilege IAM roles. Developers can deploy safely within guardrails without becoming security experts themselves.
  • Security as Code: Security policies expressed as version-controlled code, enforced automatically in CI/CD pipelines. SAST, secret scanning, and SCA checks run on every commit — immediate feedback before code reaches production.
  • Key Metrics: Mean Time to Remediate (MTTR) for vulnerabilities; vulnerability escape rate (flaws that reach production); automated coverage of high-risk modules.
Chapter 18 Defence

Runtime
Protection

Static analysis and design-time controls catch many vulnerabilities — but sophisticated attacks only manifest at runtime. This chapter covers the tools and architectures that provide defence while code is actually executing.

Web Application Firewalls (WAFs)

WAFs operate at Layer 7, inspecting HTTP/HTTPS payloads for malicious patterns. They are highly effective at blocking common OWASP Top 10 threats — SQL injection, XSS, command injection — through signature matching and anomaly detection.

WAF Strengths
  • Effective against known attack signatures (SQLi, XSS)
  • Can be deployed without code changes
  • Centralised protection for multiple applications
  • Rate limiting and DDoS absorption
  • Virtual patching for known CVEs before code fix is ready
WAF Limitations
  • Cannot understand application business logic
  • Blind to BOLA/IDOR — requests look syntactically valid
  • ~95% of API attacks come from authenticated users with valid tokens
  • Attackers encode payloads to bypass signature matching
  • Often set to monitor-only mode to avoid false-positive business impact

Runtime Application Self-Protection (RASP)

Unlike WAFs that sit at the network edge, RASP agents embed directly into the application runtime (JVM, .NET CLR, Node.js process). By intercepting internal function calls, RASP can inspect data parameters as code actually executes — seeing the internal logic flow rather than just the external HTTP surface.

✅
RASP's Contextual Advantage

When a RASP agent intercepts a SQL query being constructed, it can see whether user input is flowing into that query without parameterisation — and terminate the request before the query executes. It can differentiate between a suspicious-looking input that is harmlessly handled and an actual exploit attempt. This is particularly valuable for protecting internal microservices that have no WAF coverage.

Limitations: RASP introduces CPU and memory overhead in the production environment. It is language-specific — a Java RASP agent cannot protect a Node.js service. And it requires careful tuning to avoid false positives that interrupt legitimate business operations.

API Gateways as Policy Enforcement Points

API gateways serve as the centralised front door for all inbound traffic — the Policy Enforcement Point (PEP) in a Zero Trust architecture. They enforce coarse-grained policies that apply universally across all backend services:

  • TLS termination — centralised certificate management; decrypt, inspect, re-encrypt
  • JWT signature validation — verify token integrity before any request reaches backend code
  • Rate limiting — per-user, per-IP, per-endpoint throttling
  • Protocol translation — convert HTTP/3 to HTTP/1.1, WebSocket to REST, gRPC to REST
  • Request/response transformation — strip informational headers before responses reach clients
⚡
The Gateway's Limitation

API gateways are excellent at "macro-authorisation" — checking whether a token is valid and whether a user has the general role to access an API. But they cannot enforce "micro-authorisation" — whether User A should see Resource B specifically. BOLA/IDOR vulnerabilities live entirely in this gap. Fine-grained object-level authorisation must be implemented in the application layer itself, not delegated to the gateway.

Chapter 19 Context

Third-Party &
Supply Chain APIs

Modern applications are deeply interconnected with third-party services. This interconnection creates supply chain risk — a compromised upstream partner can become a vector for attacking every downstream consumer.

The Implicit Trust Trap

Developers apply rigorous validation to direct user input but frequently treat third-party API responses as inherently trustworthy. This asymmetry is exploitable: if a payment processor, identity provider, or data enrichment service is compromised, it can deliver malicious payloads through a channel that bypasses standard input filtering.

🚨
Experian Partner API — Business Logic Breach

A partner with legitimate authorisation to perform credit checks integrated the API into a public-facing website without adequate access controls. Unauthorised users could query the API directly, accessing millions of credit scores. The breach exploited a business logic flaw — the partner was authorised, but had created an unauthorised access path. Providers must enforce granular access controls and cannot rely on partners to implement controls correctly.

Service-to-Service Token Security

Token Forwarding Antipattern

Forwarding a user's original access token through a chain of microservices creates "overprivileged" tokens. Every service in the chain receives permissions it doesn't need. One compromised service can use the forwarded token to access all downstream resources the user could access.

Token Exchange Pattern

Each service-to-service call uses a fresh, narrowly-scoped token via OAuth 2.0 Token Exchange. Service A calls Service B with a token limited to exactly what Service B needs — not the end user's full token. Blast radius of any compromise is limited to that specific service pair.

Privilege Creep in Machine Identities

Service accounts accumulate permissions over time as requirements evolve — permissions added but never removed. Machine identities often outnumber human ones and are less frequently reviewed. Regularly audit service account permissions against actual usage patterns.

The Confused Deputy Problem

A high-privileged service (the deputy) performs an action for a user without verifying if the user has authority for it. An attacker tricks a system-level service into executing a database operation on their behalf — one they could never perform directly. The deputy's privilege is exploited to escalate the attacker's effective permissions.

Webhook Security

Webhooks are incoming HTTP requests where the receiver did not initiate the connection. Standard session authentication doesn't apply — the server must verify the sender's identity and message integrity through alternative means.

HMAC Signature Verification

The sender computes an HMAC-SHA256 of the payload using a shared secret and includes it in a header (e.g., X-Signature-256). The receiver independently computes the HMAC of the received payload and compares — if they match, the payload is authentic and unmodified.

Replay Attack Prevention

Even authentic signatures can be captured and replayed. Include a timestamp in the signed payload and reject any webhook received more than a defined "replay window" (e.g., 5 minutes) after its timestamp. Combine with a nonce for stronger protection.

🚨
Webhooks as SSRF Vectors

When an application accepts a user-supplied URL for webhook delivery and then makes requests to that URL without validation, it creates an SSRF vulnerability. Attackers configure webhook URLs pointing to internal services (e.g., http://169.254.169.254/ for cloud metadata, http://internal-admin:8080/ for internal portals). The application becomes a proxy attacking its own infrastructure. Strict allowlists of permitted webhook destination domains and outbound proxy inspection are essential mitigations.

Chapter 20 Context

API Testing &
Security Assessment

Effective security testing combines the scale of automation with the contextual intelligence of manual human expertise. Understanding when to use each, and how to combine them, maximises coverage while remaining operationally feasible.

Automated vs Manual Testing

Automated Scanning
  • Speed and scale — test thousands of inputs in minutes
  • Effective for known patterns: SQLi, XSS, known CVEs
  • Consistent coverage — won't forget to test a parameter
  • Cannot understand business intent or context
  • Will not find logic flaws: "can I skip payment step?"
  • High false-positive rates require human triage
Manual Testing
  • Human mindset understands application intent
  • Can find BOLA/BFLA through A-B user testing
  • Identifies workflow manipulation and logic bypasses
  • Slower — not feasible for every deployment
  • Tester quality determines effectiveness
  • Manual findings can tune automated tools for future scans

OpenAPI Specification as a Security Contract

The OpenAPI Specification (OAS) provides a machine-readable definition of every endpoint, parameter type, and constraint. This serves as a single source of truth for security testing and enables "design-time auditing" with linting tools like Spectral — finding missing authentication directives or unconstrained schemas before code is written.

Eliminating Shadow APIs Through Specification

Importing OAS definitions into testing tools (OWASP ZAP, Burp Suite) "seeds" the scanner with all documented endpoints — including those not linked in the UI. This ensures 100% coverage of the documented surface. For undocumented Shadow APIs, tools like Kiterunner use wordlists of common API path patterns to discover hidden endpoints through automated enumeration.

Authorization Testing Strategy

The A-B Testing Matrix for Multi-Tenant Systems

The most rigorous approach to BOLA/BFLA testing: create resources as one user (Tenant A) and attempt to access, modify, or delete them using another user's session (Tenant B). This tests both horizontal escalation (accessing peer data) and vertical escalation (lower-privilege user accessing admin functions).

Test Case Actor Expected BOLA if...
Read own resource Resource owner 200 OK N/A — baseline
Read another user's resource Different user, same role 403 Forbidden 200 returned with data
Read admin resource Standard user 403 Forbidden 200 returned with data (also BFLA)
Modify another user's resource Different user, same role 403 Forbidden 200 with modification applied
Delete another user's resource Different user, same role 403 Forbidden 200 with deletion successful

Audit vs Red Team

Compliance Audit

Policy-centric. Compares the system against a standard (PCI DSS, CIS Benchmarks). Produces point-in-time evidence of adherence to specific rules. Answers "are you following the rules?" Good for regulatory requirements; poor at testing whether those rules actually provide security.

Red Team Operation

Goal-oriented and adversarial. Uses real-world Tactics, Techniques, and Procedures (TTPs) to achieve a specific objective: "exfiltrate the crown jewels." Tests the Blue Team's ability to detect and respond to a live threat. Answers "would the rules actually protect you against a skilled attacker?"

Both approaches are necessary. Compliance proves baseline hygiene. Red teaming proves resilience. An organisation with strong audit scores but no red team experience may be compliant but trivially compromised by a skilled attacker.

Code Spaces (2014) — Credential Theft → Total Destruction

An attacker gained administrative access to Code Spaces' AWS control plane via stolen credentials and demanded payment (extortion). When the company attempted to regain access, the attacker used established persistence to delete all EC2 instances, EBS volumes, machine images, and S3 backups simultaneously. The company had no off-platform backups — it was permanently destroyed.

Root causes: Shared administrative credentials; no MFA on cloud console; all backups in the same account as production. Prevention: MFA on cloud console, separate AWS accounts for backups, cross-account backup replication, incident response plan tested in advance.

SolarWinds (2020) — Supply Chain Compromise

Attackers (nation-state) compromised SolarWinds' build pipeline, injecting SUNBURST malware into the Orion software update mechanism. The malicious code was digitally signed by SolarWinds' legitimate certificate and distributed as a trusted software update to approximately 18,000 organisations, including US government agencies.

Root causes: Insufficient build pipeline security; no code signing verification against a known-good baseline; inadequate monitoring of build environment. Prevention: Reproducible builds with cryptographic attestation, separate build environment with strict access controls, SBOM verification, behavioural monitoring of update mechanisms.

Trello (2024) — Unauthenticated API → Mass Data Scraping

A threat actor exploited an unauthenticated API endpoint to query user data by email address. By automating requests with a large list of known email addresses, the attacker extracted profile information for over 15 million Trello users. The API was designed to facilitate legitimate look-up functionality but had no authentication requirement and no rate limiting.

Root causes: Missing authentication on a public endpoint; no rate limiting; no anomaly detection for bulk enumeration. Prevention: All API endpoints must require authentication; rate limiting per IP and per authenticated user; anomaly detection for bulk sequential queries.

Uber (2022) — MFA Fatigue → Full Infrastructure Compromise

An 18-year-old attacker purchased Uber employee credentials on the dark web, then bombarded the employee with MFA push notifications until the employee approved one. With valid session access, the attacker escalated via finding a PowerShell script in a network share containing hard-coded admin credentials for PAM (Privileged Access Management). Full administrative access to AWS, GCP, Google Workspace, and internal tools was obtained.

Root causes: No MFA fatigue countermeasures (number matching); hard-coded credentials in scripts accessible to employees; insufficient lateral movement barriers. Prevention: MFA number matching, behavioural monitoring for prompt floods, secrets scanning on internal repositories, network segmentation limiting scope of credential reuse.

Operationalising Security Governance

From Gatekeepers to Guardrails

Traditional security teams act as manual review bottlenecks. Modern DevSecOps transforms the security team into providers of "secure-by-default" infrastructure — guardrails that enable developers to deploy safely within pre-approved patterns without requiring security team approval for every deployment.

Security as Code

Security policies expressed as version-controlled code artifacts. SAST rules, DAST configurations, and infrastructure policies automatically enforced with every CI/CD commit. Standards like NIST or OWASP are validated programmatically — not checked manually once a quarter.

Design-First API Governance

Using OpenAPI Specification as a formal contract before coding begins. Only operations explicitly defined in the contract are permitted — a positive security model.

API Gateway as Governance Layer

The API gateway centralises enforcement of governance policies across all microservices consistently. Rate limiting, authentication validation, logging, and version management applied uniformly — eliminating per-service implementation inconsistency that creates security gaps.

✅
The Security Programme Maturity Model

Compliance proves you followed the rules. Audits establish a baseline of cyber hygiene and satisfy regulatory requirements. Red teaming proves whether those rules actually work. Combining both approaches ensures an organisation has consistent controls and the resilience to survive a targeted attack by a skilled adversary. Compliance without resilience testing is theatre. Resilience testing without compliance is ungoverned. Both, together, are a mature security programme.

Chapter Summary — Core Principles

  • Zero Trust everywhere: Never grant trust based on network location. Every request authenticated and authorised, every time.
  • Deny by default: Access denied unless explicitly granted. Fail secure — never fail open.
  • Least privilege: Minimum permissions for every user, service, and token. Regularly audited against actual usage.
  • Shift left: Security integrated at design and development phases. The earlier a flaw is found, the cheaper it is to fix.
  • Defence in depth: Multiple independent security controls at each layer. No single control is assumed to be sufficient.
  • Inventory everything: You cannot protect what you cannot see. Complete API catalogues, SBOM, and infrastructure inventory are non-negotiable.
  • Assume breach: Design monitoring and detection assuming attackers will eventually get in. The goal is rapid detection and containment — not perfect prevention.
  • Automate security: Manual security checks don't scale. Security as code, enforced in CI/CD, is the only sustainable approach at modern development velocity.
