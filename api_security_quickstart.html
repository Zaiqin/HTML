Architecture & Threat Surface

Foundations
Monolithic

Single deployable unit. Centralized controls but "all eggs in one basket" — a breach can expose everything including DB credentials.

Microservices

Independent services via REST/gRPC. Enables micro-segmentation but each service connection needs its own auth & authz controls.

Serverless (FaaS)

Ephemeral, event-driven. Smaller malware persistence surface but vulnerable to "Denial of Wallet" via excessive invocations.

Key Insight

Traditional AppSec tools (SAST/DAST) are insufficient for APIs. Network firewalls inspect IPs/ports but cannot prevent attacks targeting application logic. APIs expand the attack surface with direct paths to sensitive data.

Authentication & Authorization

Identity
Core Distinction

Authentication = "Are you who you say you are?"  |  Authorization = "What are you allowed to do?" — Coupling them leads to privilege escalation. Successful AuthN does NOT mean open access.

MFA Strength Ranking
    1
    FIDO2 / Passkeys
    Cryptographically bound to domain. Technically un-phishable — device refuses if domain doesn't match.
    Strongest
    2
    TOTP (Authenticator App)
    Time-based codes. Immune to SIM swap but vulnerable to real-time AiTM phishing proxies.
    Good
    3
    Push Notifications
    Vulnerable to MFA Fatigue (prompt bombing). Mitigated by number matching.
    Weak
    4
    SMS / Voice
    Vulnerable to SIM swapping and SS7 interception. Deprecated by NIST in 2016.
    Weakest
JWT — Critical Validations
  • alg — hardcode expected algorithm server-side; never trust client header
  • none algorithm — must be rejected; strip-signature attack
  • exp — always validate expiry; missing check enables replay attacks
  • aud — verify audience; a token for Service A shouldn't work on Service B
  • iss — verify issuer; prevents cross-tenant token replay
  • Payload is not encrypted — base64url only; never store PII without JWE
OAuth 2.0 Flow Guide
Auth Code + PKCE

Gold standard for user-facing apps. Back-channel token exchange. Defeats code interception.

Client Credentials

Machine-to-machine only. No user session. Service authenticates with its own ID + secret.

⚠ Implicit Flow and ROPC are deprecated — both expose credentials or tokens directly.

OWASP API Security Top 10

API-Specific
    01
    BOLA / IDOR — Broken Object Level Authorization
    Changing /orders/1001 → /orders/1002 accesses another user's data. Most critical API flaw — logic error, not code error. Every resource request must verify ownership.
    Critical
    02
    Broken Authentication
    Weak credential handling, no rate limiting, JWT "none" algorithm, missing expiry validation.
    Critical
    03
    BOPLA — Broken Object Property Level Auth
    Combines Mass Assignment (injecting is_admin:true) and Excessive Data Exposure (API returns full object; client filters). Never trust client-supplied fields blindly.
    Critical
    04
    Unrestricted Resource Consumption
    No rate limits → credential stuffing, scraping, DoS. In cloud: "Denial of Wallet" — auto-scaling runs up massive bills.
    High
    05
    BFLA — Broken Function Level Authorization
    Standard user accesses admin endpoints via forced browsing or HTTP method switching (GET → DELETE). Role must be checked at every function, not just login.
    High
    06
    Unrestricted Access to Sensitive Business Flows
    Scalper bots, coupon abuse, fake accounts — technically valid requests that violate business intent.
    High
    07
    Server-Side Request Forgery (SSRF)
    API fetches attacker URL → hits internal services or cloud metadata at 169.254.169.254 to steal IAM credentials. Capital One breach vector.
    High
    08
    Security Misconfiguration
    GraphQL introspection left on, CORS wildcard with credentials, verbose stack traces, default API keys.
    Medium
    09
    Improper Inventory Management
    Shadow APIs (undocumented) and Zombie APIs (deprecated but still live) — unpatched, unmonitored backdoors.
    Medium
    10

Container & K8s Security
  • Scan images with Trivy/Clair before pushing to registry
  • Run as non-root — limits container breakout blast radius
  • RBAC — a web server pod shouldn't list cluster secrets
  • mTLS via service mesh (Istio/Linkerd) for East-West traffic
  • Secure the API server — 2018 Tesla breach was an exposed dashboard
  • Runtime protection with Falco/eBPF for anomaly detection
Zero Trust Mandate

The "castle-and-moat" model is dead. Every service-to-service call must be authenticated. Internal network location grants no trust. Use mTLS + token-based service authorization everywhere.

07

Secure API Design Principles

Design
Least Privilege

Tokens scoped to minimum. Read-only apps get read-only tokens. Backend services don't forward over-privileged tokens. Whitelist updatable fields.

Deny by Default

All access denied unless explicitly granted. If a config fails, result is locked state. Never "fail open."

Filter at Backend

Never rely on UI to hide sensitive fields. API returns only what the client legitimately needs. Treat all client input as untrusted.

Generic Errors

Return 400/500 to client. Log details internally. Consistent response size/time to prevent side-channel attacks.

Idempotency & Anti-Replay

Use nonces or ETags to prevent replayed requests. ETags reject updates if resource changed since last fetch (412 Precondition Failed).

API Lifecycle

Maintain an API catalog. Use Sunset/Deprecation headers. Retire zombie APIs via gateway automation before they become backdoors.

08

Client-Side Security

Browser
Token Storage Trade-offs
LocalStorage

Accessible via JS → XSS can exfiltrate tokens silently. Avoid for sensitive tokens.

HttpOnly Cookie

JS can't read it → XSS-safe. But browser auto-sends it → CSRF risk. Mitigate with SameSite + CSRF tokens.

BFF Pattern (Best)

Token management stays server-side. Browser gets encrypted HttpOnly+SameSite cookies. Credentials never touch the browser directly.

Security Headers Checklist
  • HSTS — force HTTPS; prevent SSL stripping
  • CSP — restrict script sources; block inline + eval; use frame-ancestors
  • X-Content-Type-Options: nosniff — prevent MIME sniffing XSS
  • Referrer-Policy — prevent token leakage in Referer header
  • SRI on third-party scripts — hash verification prevents CDN-compromise attacks
Clickjacking

Invisible iframes redirect user clicks to perform actions on a target site. Use frame-ancestors 'none' in CSP (supersedes X-Frame-Options).

Secrets Management

Credentials
The Golden Rule

A leaked API key in a cloud environment = username + password + MFA bypass. Static, long-lived credentials are the root cause of most high-profile breaches. Favor short-lived credentials (STS tokens) over static IAM keys.

Where Secrets Leak
  • Hardcoded in source code — visible to all repo access; permanent in git history
  • Config/backup files — .env, wp-config.php, .bak files in web root
  • Client-side code — JS bundles and mobile APKs are decompilable
  • Verbose logs — Authorization headers, tokens written to debug logs
  • Environment variables — exposed via /proc or command injection
Managed Secret Stores
AWS Secrets Manager / HashiCorp Vault

Centralized encrypted store. Apps retrieve secrets at runtime via API — never touch disk. Full audit trail. Automated rotation limits blast radius of any leak.

Revocation in Distributed Systems

JWTs remain valid until expiry even after password changes. Use short-lived tokens (5–15 min) + blocklists for compromised token IDs.

Logging, Monitoring & Detection

Detection
What to Log (The Five Ws)
  • Who — authenticated user ID, session ID
  • What — action taken, resource accessed (CRUD on sensitive data)
  • Where — source IP, User-Agent
  • When — precise timestamp (UTC)
  • Result — success/failure, HTTP status code
  • Never log — passwords, tokens, credit card numbers (PII minimization)
Detection Signals
404 burst = recon High 403s = privilege probe Impossible travel sqlmap User-Agent Consistent request timing = bot SYN scans Beaconing pattern New device + geo
Dwell Time Reality

Attackers average 200+ days in networks before detection. Most organizations are notified by third parties (FBI, fraud processors), not their own monitoring. Centralize logs in a SIEM — avoid "data graves."

SAST

Scans source code without running it. Catches injection flaws early, points to exact line. High false-positive rate; misses runtime issues.

DAST

Black-box testing against running app. Finds runtime issues (insecure cookies, auth flows). Late in lifecycle; can't pinpoint code location.

IAST

Agent embedded in runtime. Gets code visibility of SAST + runtime context of DAST. Much lower false-positive rate.

Governance, Standards & Compliance

Governance
NIST SP 800-207 (Zero Trust)

Every request evaluated by a Policy Decision Point. Access never based on network location. "Never trust, always verify."

GDPR — Data Protection by Design

Privacy built into SDLC from day one. APIs must support data minimization and Right to Erasure across distributed systems.

PCI DSS v4.0

Mandates MFA for privileged access. Requires pen testing specifically targeting business logic flaws — not just automated scanning.

ISO 27001 / 27036

ISMS framework mandating least privilege (A.9) and network segregation (A.13). ISO 27036 specifically covers supplier/third-party API risk.

The Operational Shift

Security teams must move from Gatekeepers (manual bottlenecks) to Guardrails (secure-by-default infrastructure). Treat security policies as code — version-controlled, automated, enforced at every CI/CD commit. The API Gateway is your central choke point for consistent enforcement across microservices.
